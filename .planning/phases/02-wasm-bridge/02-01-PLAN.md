**Step 4: Create src/wasm_api.rs — the WASM interop surface**

Per D-01: Export the `Game` struct directly with `#[wasm_bindgen]` as an opaque handle. JS interacts through methods.

Create `src/wasm_api.rs` with:

```rust
use wasm_bindgen::prelude::*;
use crate::board::{Game, GameStatus, Player};
use crate::ai::get_computer_move;

/// Initialize panic hook for better error messages in browser console.
#[wasm_bindgen(start)]
pub fn init_panic_hook() {
    console_error_panic_hook::set_once();
}

/// Opaque game handle exported to JavaScript (per D-01).
#[wasm_bindgen]
pub struct WasmGame {
    inner: Game,
}

#[wasm_bindgen]
impl WasmGame {
    /// Create a new game. X goes first.
    #[wasm_bindgen(constructor)]
    pub fn new() -> WasmGame {
        WasmGame { inner: Game::new() }
    }

    /// Attempt to place the current player's piece at position (0-8).
    /// Returns true on success, false on failure. Check last_error() for details.
    pub fn make_move(&mut self, position: usize) -> bool {
        self.inner.make_move(position).is_ok()
    }

    /// Get the board state as a flat array of 9 u8 values (per D-03):
    /// 0 = empty, 1 = X, 2 = O
    pub fn get_board(&self) -> Vec<u8> {
        self.inner.cells().iter().map(|cell| match cell {
            None => 0u8,
            Some(Player::X) => 1u8,
            Some(Player::O) => 2u8,
        }).collect()
    }

    /// Get the current player: 1 = X, 2 = O
    pub fn current_player(&self) -> u8 {
        match self.inner.current_player() {
            Player::X => 1,
            Player::O => 2,
        }
    }

    /// Get game status as a string (per D-02 — agent's discretion):
    /// "playing", "won", or "draw"
    pub fn get_status(&self) -> String {
        match self.inner.status() {
            GameStatus::InProgress => "playing".to_string(),
            GameStatus::Won { .. } => "won".to_string(),
            GameStatus::Draw => "draw".to_string(),
        }
    }

    /// Get the winner: 0 = no winner, 1 = X, 2 = O
    pub fn get_winner(&self) -> u8 {
        match self.inner.status() {
            GameStatus::Won { winner, .. } => match winner {
                Player::X => 1,
                Player::O => 2,
            },
            _ => 0,
        }
    }

    /// Get winning positions as a 3-element array, or empty vec if no winner.
    /// Returns the 3 cell indices (0-8) that form the winning line.
    pub fn get_winning_positions(&self) -> Vec<usize> {
        match self.inner.status() {
            GameStatus::Won { positions, .. } => positions.to_vec(),
            _ => vec![],
        }
    }

    /// Ask the AI to make a move. Returns the chosen position (0-8),
    /// or 255 if the game is already over.
    pub fn computer_move(&mut self) -> u8 {
        match get_computer_move(&self.inner) {
            Some(pos) => {
                let _ = self.inner.make_move(pos);
                pos as u8
            }
            None => 255,
        }
    }

    /// Reset the game to initial state.
    pub fn reset(&mut self) {
        self.inner = Game::new();
    }
}
```

Key design decisions:
- Per D-01: `WasmGame` wraps `Game` as an opaque handle with methods
- Per D-02 (agent's discretion): `GameStatus` enum is exposed via separate getter methods (`get_status()` returns string, `get_winner()` returns u8, `get_winning_positions()` returns Vec) — avoids serde dependency, keeps the boundary clean with scalar types
- Per D-03: Board is returned as `Vec<u8>` with values 0/1/2 — maps to JS `Uint8Array`
- `computer_move()` both computes AND applies the move (convenience for JS side — single call instead of get_move + make_move)
- Return value 255 signals "no move" (game over) — unambiguous since valid positions are 0-8

        import init, { WasmGame } from './pkg/tic_tac_toe.js';

            assert(board.length === 9, 'Board has 9 cells');

            const board2 = game.get_board();
            assert(board2[4] === 1, 'Cell 4 is now X (1)');

            // Test 7: Player switched to O
            assert(game.current_player() === 2, 'Current player is now O (2)');

            // Test 8: Computer makes a move
            const compPos = game.computer_move();
            assert(compPos >= 0 && compPos <= 8, `Computer chose position ${compPos} (0-8)`);
            const board3 = game.get_board();
            assert(board3[compPos] === 2, `Cell ${compPos} is now O (2)`);

            // Test 9: Player back to X after computer move
            assert(game.current_player() === 1, 'Back to X after computer move');

            // Test 10: Invalid move (occupied cell) returns false
            const badMove = game.make_move(4); // cell 4 already has X
            assert(badMove === false, 'make_move on occupied cell returns false');

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Rust→WASM | Compiled Rust code runs in browser sandbox — no filesystem/network access |
| JS→WASM | JavaScript calls into WASM via wasm-bindgen glue — type-safe boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | WASM input validation | accept | Game.make_move() already validates bounds (0-8), occupied cells, and game-over state in Rust — invalid inputs return errors, no undefined behavior possible |
| T-02-02 | Denial of Service | Minimax computation | accept | 9-cell board has bounded game tree (~9! max nodes) — minimax cannot hang. No network calls, no user-controlled recursion depth |
| T-02-03 | Information Disclosure | Game state | accept | No sensitive data — game board state is inherently public (visible on screen). No PII, no auth, no secrets |
</threat_model>
