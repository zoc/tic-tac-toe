## Releasing

This project uses GitHub Actions to automatically build and publish multi-platform Docker images to Docker Hub on every version tag push.

### Prerequisites

The repository maintainer must configure two GitHub settings:

**Secret: DOCKERHUB_TOKEN**
- Navigate to: Repository → Settings → Secrets and variables → Actions → New repository secret
- Name: `DOCKERHUB_TOKEN`
- Value: Docker Hub Personal Access Token (create at Docker Hub → Account Settings → Security → New Access Token with Read & Write scope)

**Variable: DOCKERHUB_USERNAME**
- Navigate to: Repository → Settings → Secrets and variables → Actions → Variables → New repository variable
- Name: `DOCKERHUB_USERNAME`
- Value: Your Docker Hub username (e.g., `johndoe`)

### Creating a release

1. **Tag the release:**
   ```bash
   git tag v1.3.0
   git push origin v1.3.0
   ```

2. **Monitor the build:**
   - GitHub Actions tab shows "Docker" workflow running
   - Build takes 5-8 minutes
   - All steps must show green checkmarks

   - Both `linux/amd64` and `linux/arm64` architectures shown

### Workflow details

**Workflow file:** `.github/workflows/docker.yml`
**Trigger:** Git tags matching `v*.*.*` (e.g., v1.2.0, v1.3.0)
**Platforms:** linux/amd64, linux/arm64 (multi-arch manifest created automatically)
**Cache:** GitHub Actions cache (speeds up subsequent builds)
**Build time:** ~5-8 minutes with warm cache

**Technical notes:**
- Rust compilation runs natively (uses `--platform=$BUILDPLATFORM` in Dockerfile to avoid QEMU)
- WASM bytecode is platform-neutral (identical output on both architectures)
- Only the nginx:alpine serve stage cross-compiles under QEMU
- Multi-arch manifest means users automatically get the correct architecture for their platform

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions → Docker Hub | Workflow authenticates with token, pushes images to public registry |
| Git tags → GitHub Actions | Public tags trigger workflow; anyone with write access can create releases |
| Docker Hub → end users | Public images; anyone can pull without authentication |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01 | Spoofing | DOCKERHUB_TOKEN secret | mitigate | Token stored in GitHub Secrets (encrypted at rest, redacted in logs). Only repository maintainers can access. Rotation recommended every 90-180 days. |
| T-11-02 | Tampering | Docker images on Docker Hub | accept | Images are publicly writable by authenticated token holder. Token compromise = image tampering risk. Mitigation: rotate tokens regularly, monitor Docker Hub for unexpected tags. Future: add image signing (DIST-03 in Phase 12+). |
| T-11-03 | Repudiation | Workflow execution | accept | GitHub Actions logs all workflow runs with timestamps, triggering user, and commit SHA. Logs retained for 90 days. Sufficient audit trail for this use case. |
| T-11-04 | Information Disclosure | Workflow logs | accept | Logs are public (repo is public). No sensitive data exposed — secrets are redacted automatically by GitHub Actions. Docker Hub username is intentionally public. |
| T-11-05 | Denial of Service | Docker Hub rate limits | accept | 50 pushes/day for personal accounts. This project releases <5 times/month. Well within limits. If hit: wait 24h or upgrade to paid Docker Hub plan. |
| T-11-06 | Elevation of Privilege | Compromised GitHub Actions runner | accept | Runners are ephemeral, isolated, and managed by GitHub. Supply chain risk from Actions marketplace exists but is mitigated by using official Docker actions (@docker org) at pinned major versions. |

## Security Notes

**ASVS Level 1 compliance:**
- V14.2.1 (Dependency management): Using official Docker actions at pinned major versions (@v3, @v6)
- V14.2.3 (Known vulnerabilities): GitHub Dependabot monitors workflow action versions
- V2.2.1 (Secrets): DOCKERHUB_TOKEN stored in GitHub Secrets, never in code or logs

**Out of scope for Phase 11:**
- Image signing / provenance (Phase 12+ consideration)
- Vulnerability scanning in CI (no security requirements in v1.3)
- Branch protection rules (assumes trusted maintainer for personal project)
</threat_model>
