**Domain:** Rust/WASM browser game (tic-tac-toe)
**Researched:** 2026-04-12

The architecture follows the canonical Rust/WASM pattern established by the official Rust WASM Book (Game of Life tutorial): **game state and logic live in Rust/WASM; rendering and user interaction live in JavaScript/CSS**. The wasm-bindgen generated JS glue layer bridges the two.

### Component Responsibilities

| Component | Responsibility | Typical Implementation |
|-----------|----------------|------------------------|
| **Board Logic** | Validates moves, detects wins/draws, manages cell state | Rust struct with `#[wasm_bindgen]`, flat array `[u8; 9]` or `Vec<Cell>` |
| **AI Engine** | Computes computer's move using minimax with controlled imperfection | Rust module, called by game state machine, returns position index |
| **Game State Machine** | Manages turns, game phase (playing/won/draw), score tracking | Rust struct exposed via `#[wasm_bindgen]`, owns Board and AI |
| **JS Glue Layer** | Auto-generated FFI bridge between Rust exports and JS imports | `wasm-pack build --target web` output in `pkg/` directory |
| **Renderer** | Draws game board, pieces, highlights to screen | Vanilla JS manipulating DOM/CSS (grid of divs or `<canvas>`) |
| **Event Handler** | Captures user clicks on cells, maps to board positions | JS event listeners on grid cells, calls WASM functions |
| **Animation/UI Controller** | Manages CSS transitions, score display, new game flow | JS + CSS animations/transitions, updates DOM state |

## Recommended Project Structure

```

├── src/
│   ├── lib.rs              # WASM entry point, #[wasm_bindgen] exports
│   ├── board.rs            # Board struct, move validation, win detection
│   ├── ai.rs               # Minimax AI with random mistake injection
│   └── game.rs             # Game state machine (turns, scores, phases)

├── www/                    # Frontend (served as static files)

│   └── index.js            # WASM init, event handling, rendering
├── pkg/                    # Generated by wasm-pack build (gitignored)
│   ├── tic_tac_toe_bg.wasm
```

### Structure Rationale

- **`src/` (Rust):** Split by domain concern — board, AI, and game state are independent modules that compose cleanly. `lib.rs` is the thin `#[wasm_bindgen]` export surface, keeping FFI concerns separate from logic.
- **`www/` (frontend):** Vanilla HTML/CSS/JS — no framework needed for a single-screen game. Keeps the project minimal and avoids build tool complexity on the JS side.
- **`pkg/` (generated):** Output of `wasm-pack build --target web`. Gitignored. Contains the compiled `.wasm` binary plus generated JS/TS glue code.
- **`tests/`:** Rust unit tests run natively with `cargo test` — no browser needed to verify game logic. This is a major advantage of the split architecture.

## Architectural Patterns

### Pattern 1: Opaque Handle Pattern (WASM ↔ JS boundary)

**What:** Rust structs marked with `#[wasm_bindgen]` are exposed to JavaScript as opaque handles (pointers into WASM linear memory). JS never directly reads struct internals — it calls exported methods that return simple values (numbers, strings, booleans).

**When to use:** Always, for the primary game state. This is the canonical Rust/WASM interop pattern.

**Trade-offs:**
- PRO: Minimizes data copying across the FFI boundary
- PRO: Rust owns all game state — no sync issues
- CON: JS cannot inspect state without calling a method — every query is a function call

**Example:**
```rust
#[wasm_bindgen]
pub struct Game {
    board: [Cell; 9],
    current_turn: Player,
    status: GameStatus,
    scores: Scores,
}

#[wasm_bindgen]
impl Game {
    pub fn new() -> Game { /* ... */ }
    pub fn make_move(&mut self, position: u8) -> bool { /* ... */ }
    pub fn get_cell(&self, position: u8) -> u8 { /* ... */ }
    pub fn get_status(&self) -> u8 { /* ... */ }
    pub fn get_winner_line(&self) -> Option<Vec<u8>> { /* ... */ }
}
```

```javascript
import init, { Game } from './pkg/tic_tac_toe.js';

await init();
const game = Game.new();
game.make_move(4);  // Human plays center
// JS never sees Game's internal struct — only calls methods
```

### Pattern 2: Scalar Return Values Across FFI

**What:** Return simple numeric types (u8, i32, bool) from WASM functions rather than complex objects. Encode game states as integer constants. Avoid returning strings or serialized JSON for frequently-called operations.

**When to use:** For all game state queries that happen on every render cycle or user interaction.

**Trade-offs:**
- PRO: Zero-copy, zero-allocation — fastest possible FFI crossing
- PRO: No serde dependency needed
- CON: Requires decoding constants on the JS side (e.g., `0 = Empty, 1 = X, 2 = O`)

**Example:**
```rust
#[wasm_bindgen]
#[repr(u8)]
#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub enum Cell {
    Empty = 0,
    X = 1,
    O = 2,
}

#[wasm_bindgen]
#[repr(u8)]
pub enum GameStatus {
    Playing = 0,
    WonX = 1,
    WonO = 2,
    Draw = 3,
}
```

```javascript
const CELL = { EMPTY: 0, X: 1, O: 2 };
const STATUS = { PLAYING: 0, WON_X: 1, WON_O: 2, DRAW: 3 };

function renderBoard(game) {
    for (let i = 0; i < 9; i++) {
        const cell = game.get_cell(i); // returns u8, no allocation
        // render based on cell value
    }
}
```

### Pattern 3: JS Owns the Render Loop, WASM Owns the State

**What:** JavaScript controls when rendering happens and when to query game state. Rust/WASM is a pure state machine — it receives commands (make a move) and answers queries (what's in cell 5? who won?). It never initiates rendering or DOM manipulation.

**When to use:** Always for Rust/WASM browser games. This is the established pattern from the official Rust WASM Book.

**Trade-offs:**
- PRO: Clean separation of concerns — Rust doesn't need web-sys for DOM
- PRO: JS handles animations/transitions naturally with CSS and requestAnimationFrame
- PRO: Rust game logic is fully testable without a browser
- CON: Cannot use Rust-side rendering frameworks (not needed for this project)

**Example flow:**
```javascript
// JS controls the entire interaction cycle
cell.addEventListener('click', () => {
    const success = game.make_move(cellIndex);  // Tell WASM
    if (success) {
        renderBoard(game);                       // Query WASM, update DOM
        if (game.get_status() === STATUS.PLAYING) {
            // Computer's turn — WASM computes, JS renders result
            game.computer_move();
            renderBoard(game);
            checkGameEnd(game);
        }
    }
});
```

## Data Flow

### Move Flow (Human Turn)

```
User clicks cell
    ↓
JS Event Handler (maps click → position index 0-8)
    ↓
game.make_move(position)  →  [FFI boundary]  →  Rust: validate move,
    ↓                                            update board, check win,
    ↓                                            switch turn
    ↓                                            ← returns bool (success)
JS reads game state via get_cell()/get_status()
    ↓
JS updates DOM (render piece, check for win/draw)
    ↓
If game still playing → trigger computer turn
```

### Move Flow (Computer Turn)

```
JS calls game.computer_move()
    ↓
[FFI boundary]  →  Rust AI Engine:
                    1. Run minimax on current board
                    2. Roll random chance for "mistake"
                    3. If mistake → pick suboptimal move
                    4. Place O on board
                    5. Check for win/draw
                    ← returns u8 (position played)
    ↓
JS reads updated state
    ↓
JS animates computer's piece placement
    ↓
JS checks game.get_status() for win/draw
    ↓
If won/draw → show result, update scores, offer new game
```

### New Game Flow

```
User clicks "New Game"
    ↓
JS calls game.reset()
    ↓
[FFI boundary]  →  Rust: clear board, set turn to X,
                    set status to Playing
                    (scores preserved)
    ↓
JS clears all cell displays
JS re-enables click handlers
```

### Key Data Flows

1. **Human move:** Click → JS event → `game.make_move(pos)` → Rust validates/updates → JS re-renders from state queries
2. **Computer move:** JS calls `game.computer_move()` → Rust runs AI + updates board → JS reads position + re-renders
3. **State query:** JS calls `game.get_cell(i)` / `game.get_status()` → Rust returns u8 scalar → JS interprets constant
4. **Win highlight:** JS calls `game.get_winner_line()` → Rust returns winning positions (e.g., `[0,1,2]`) → JS highlights those cells
5. **Score tracking:** JS calls `game.get_score_x()` / `game.get_score_o()` / `game.get_draws()` → Rust returns u32 → JS updates score display

## Scaling Considerations

This is a single-player browser game with no server component. Traditional scaling is irrelevant. The relevant "scaling" concerns are:

| Concern | Impact | Approach |
|---------|--------|----------|
| WASM binary size | Affects initial load time | Use `wasm-opt` (included in wasm-pack), `lto = true` in release profile, avoid heavy dependencies |
| AI computation time | Minimax on 3x3 is trivially fast | Full minimax tree for tic-tac-toe has ~255k nodes max — runs in microseconds even unoptimized |
| Memory footprint | Negligible for tic-tac-toe | Board is 9 bytes. Entire game state fits in a cache line |
| Browser compatibility | Must work on modern browsers | `--target web` with ES modules works on all modern browsers (Chrome 61+, Firefox 60+, Safari 11+, Edge 79+) |

### Scaling Priorities

1. **Only bottleneck:** WASM binary size. Keep dependencies minimal. A bare wasm-bindgen tic-tac-toe should compile to ~20-50KB `.wasm` (after wasm-opt). Adding serde or other heavy crates can bloat this 5-10x.
2. **Non-issue:** CPU/memory. Tic-tac-toe minimax is O(9!) = 362,880 leaf nodes in the worst case — effectively instant.

## Anti-Patterns

### Anti-Pattern 1: Serializing Game State as JSON Across FFI

**What people do:** Use `serde_json` + `serde-wasm-bindgen` to serialize the entire board state to a JS object on every render.
**Why it's wrong:** Adds ~50-100KB to WASM binary size (serde + serde_json). Creates unnecessary allocations. For a 9-cell board queried by index, scalar returns are 100x more efficient.
**Do this instead:** Return individual cell values as `u8` via `get_cell(position)`. Return game status as `u8` enum. Only use serde if you need to pass complex, irregular data (you don't for tic-tac-toe).

### Anti-Pattern 2: Using web-sys for DOM Manipulation from Rust

**What people do:** Import `web-sys` to manipulate the DOM directly from Rust — creating elements, setting styles, handling events from Rust code.
**Why it's wrong:** Dramatically increases WASM binary size (web-sys is huge). Makes Rust code untestable without a browser. Mixes concerns. JavaScript/CSS already handle DOM beautifully.
**Do this instead:** Keep Rust as a pure state machine. Let JavaScript handle all rendering and DOM interaction. The FFI boundary should be a thin command/query API.

### Anti-Pattern 3: Putting UI State in Rust

**What people do:** Track animation state, button enabled/disabled, display text in the Rust struct.
**Why it's wrong:** Crosses the FFI boundary for purely visual concerns. Makes Rust code coupled to the specific UI implementation.
**Do this instead:** Rust tracks game state only (board, turns, scores, win/draw). JavaScript derives all UI state from game state queries. Example: "is cell clickable?" = JS checks `game.get_status() === PLAYING && game.get_cell(i) === EMPTY`.

### Anti-Pattern 4: Using a Bundler When You Don't Need One

**What people do:** Set up Webpack/Vite/Rollup just to load a single WASM module.
**Why it's wrong:** Adds build complexity, dependencies, and a second toolchain for no benefit in a simple project.
**Do this instead:** Use `wasm-pack build --target web` which produces ES modules loadable directly in `<script type="module">`. Serve with any static file server (e.g., `python3 -m http.server`). Only add a bundler if you later need npm dependencies or code splitting.

## Integration Points

### External Services

None. This is a fully client-side browser game with no backend, no APIs, no databases.

### Internal Boundaries

| Boundary | Communication | Notes |
|----------|---------------|-------|
| **JS ↔ WASM** | Function calls via wasm-bindgen glue | The critical boundary. Keep it thin: commands in (make_move, computer_move, reset), queries out (get_cell, get_status, get_scores). All values are scalars (u8, u32, bool). |
| **lib.rs ↔ game.rs** | Direct Rust module imports | `lib.rs` is the `#[wasm_bindgen]` export surface. `game.rs` contains the Game struct. `lib.rs` re-exports Game. |
| **game.rs ↔ board.rs** | Game struct owns Board struct | Board handles move validation, win detection. Game handles turn management, AI delegation, scores. |
| **game.rs ↔ ai.rs** | Game calls `ai::compute_move(&board)` | AI is a pure function: takes board state, returns position. Stateless. Easy to test independently. |
| **JS Renderer ↔ JS Event Handler** | Shared DOM references | Both operate on the same DOM elements. Event handler triggers state change + re-render. Keep rendering in a single `renderBoard()` function for consistency. |

## Build Order (Dependency Chain)

The components have clear dependencies that dictate build order:

```
Phase 1: board.rs (zero dependencies)
    ↓ depends on
Phase 2: ai.rs (depends on board for state representation)
    ↓ depends on
Phase 3: game.rs + lib.rs (depends on board + ai, adds wasm-bindgen exports)
    ↓ depends on
Phase 4: www/ (depends on compiled WASM pkg/ output)
    ↓ depends on
Phase 5: Polish (depends on working game loop)
```

**Detailed build order rationale:**

1. **Board first** — It's the foundation. Cell type, board struct, move validation, win detection. Can be fully unit-tested in pure Rust with `cargo test`. No WASM or browser needed.

2. **AI second** — Depends on board representation to evaluate positions. Minimax operates on Board. Also fully testable in pure Rust — feed it board states, assert it picks sensible moves.

3. **Game state + WASM exports third** — Composes board and AI into the Game struct. Adds `#[wasm_bindgen]` annotations to expose the public API. First time you need `wasm-pack build`. Test the FFI surface.

4. **Frontend fourth** — Now that the WASM module is built and exposes a working API, build the HTML/CSS/JS frontend. Wire up click handlers, rendering, and the game loop. This is where you see it working in a browser for the first time.

5. **Polish last** — Animations, win highlighting, score display, visual flair. Depends on the complete working game loop being in place. Purely additive — CSS transitions, color schemes, layout refinement.
