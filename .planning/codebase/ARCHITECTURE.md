# Architecture

**Analysis Date:** 2026-04-14

## Pattern Overview

**Overall:** Split WASM + Vanilla JS architecture with a thin interop boundary

**Key Characteristics:**
- Rust owns all game logic (state, rules, AI); JavaScript owns all rendering and events
- A single opaque struct (`WasmGame`) crosses the WASM boundary — JS never sees internal Rust types
- No JavaScript framework; DOM is mutated directly and incrementally by `src/main.js`
- UI changes (CSS, animations, sounds) require zero Rust recompilation
- Dual entrypoints: browser game via WASM (`src/lib.rs`), CLI game via binary (`src/main.rs`)

## Layers

**Game Core (Rust — pure logic):**
- Purpose: Board state, move validation, win/draw detection
- Location: `src/board.rs`
- Contains: `Game` struct, `Player` enum, `GameStatus` enum, `WIN_LINES` constant
- Depends on: nothing (no external crates)
- Used by: `src/ai.rs`, `src/wasm_api.rs`, `src/main.rs`

**AI Engine (Rust — pure logic):**
- Purpose: Computer move selection via minimax with deliberate mistake injection
- Location: `src/ai.rs`
- Contains: `get_computer_move()`, `minimax()`, `check_winner()`, `WIN_LINES` (local copy)
- Depends on: `src/board.rs`, `rand` crate
- Used by: `src/wasm_api.rs`, `src/main.rs`

**WASM API Layer (Rust — interop boundary):**
- Purpose: Expose game logic to JavaScript via `#[wasm_bindgen]`; translate Rust types to JS-safe primitives
- Location: `src/wasm_api.rs`
- Contains: `WasmGame` struct (opaque JS handle wrapping `Game`), `init_panic_hook()`
- Depends on: `src/board.rs`, `src/ai.rs`, `wasm-bindgen`, `console_error_panic_hook`
- Used by: JS frontend via generated `pkg/tic_tac_toe.js`

**Generated WASM Glue (build artifact — do not edit):**
- Purpose: JS ↔ WASM memory bridge auto-generated by `wasm-pack`
- Location: `pkg/tic_tac_toe.js`, `pkg/tic_tac_toe_bg.wasm`, `pkg/tic_tac_toe.d.ts`
- Contains: Memory management, type marshalling (strings, arrays), `WasmGame` JS class
- Not committed (in `.gitignore`); regenerated by `wasm-pack build --target web --release`

**Frontend Controller (JavaScript):**
- Purpose: DOM rendering, event handling, score persistence, async computer-turn orchestration
- Location: `src/main.js`
- Contains: `main()` entry point, `renderBoard()`, `handleCellClick()`, `handleGameOver()`, `resetGame()`
- Depends on: `../pkg/tic_tac_toe.js` (WASM), `./audio.js`

**Audio Module (JavaScript):**
- Purpose: Synthesised sound effects via Web Audio API; mute toggle with localStorage persistence
- Location: `src/audio.js`
- Contains: `sounds` export object, `toggleMute()`, `isMuted()`
- Depends on: Web Audio API (browser built-in)

**HTML Shell:**
- Purpose: Static DOM skeleton; JS injects the 9 cell divs at runtime
- Location: `index.html`
- The board `<div id="board">` starts empty; cells are created by `main.js` on init

**Styles:**
- Purpose: All visual presentation including animations, win-line overlay, responsive sizing
- Location: `src/style.css`
- CSS custom properties (`--bg`, `--accent`, etc.) provide theming; dark/light auto-switches via `prefers-color-scheme`

## WASM Boundary

### What Rust Exports (via `#[wasm_bindgen]`)

All exports live in `src/wasm_api.rs` and surface through `pkg/tic_tac_toe.js`:

| Rust Method | JS Signature | Return Type | Notes |
|---|---|---|---|
| `WasmGame::new()` | `new WasmGame()` | `WasmGame` | Constructor |
| `WasmGame::make_move(pos)` | `game.make_move(index)` | `boolean` | Returns false on invalid move |
| `WasmGame::get_board()` | `game.get_board()` | `Uint8Array[9]` | 0=empty, 1=X, 2=O |
| `WasmGame::current_player()` | `game.current_player()` | `number` | 1=X, 2=O |
| `WasmGame::get_status()` | `game.get_status()` | `string` | `"playing"`, `"won"`, `"draw"` |
| `WasmGame::get_winner()` | `game.get_winner()` | `number` | 0=none, 1=X, 2=O |
| `WasmGame::get_winning_positions()` | `game.get_winning_positions()` | `Uint32Array[3]` | Empty if no winner |
| `WasmGame::computer_move()` | `game.computer_move()` | `number` | Returns 255 sentinel if game over |
| `WasmGame::reset()` | `game.reset()` | `void` | Resets board in-place |

**Design principle:** Internal Rust types (`Player`, `GameStatus`, `Game`) never cross the boundary. `WasmGame` translates everything to primitives (integers, strings, typed arrays) to avoid serde overhead and keep the contract simple.

### How JS Initializes WASM

```js
// src/main.js
import init, { WasmGame } from '../pkg/tic_tac_toe.js';

async function main() {
  await init();           // fetches + instantiates the .wasm binary
  game = new WasmGame();  // allocates Rust struct in WASM linear memory
  // ...
}
```

The `init()` call is asynchronous (network fetch of `.wasm`). All WASM calls after `init()` resolves are synchronous.

## Data Flow

### User Move → DOM Update

1. User clicks a cell div (or presses Enter/Space via keyboard)
2. `boardEl` click/keydown event fires → `handleCellClick(event)` in `src/main.js`
3. Guard checks: `isProcessing`, `game.get_status() !== 'playing'`, `game.current_player() !== PLAYER_X`
4. `cell.dataset.index` → integer position (0–8)
5. `game.make_move(index)` → WASM call → `WasmGame::make_move()` → `Game::make_move()` mutates board in WASM linear memory → returns `boolean`
6. If move accepted: `sounds.move()`, then `renderBoard()` polls `game.get_board()` → `Uint8Array[9]` → incremental DOM patch (only changed cells updated)
7. `game.get_status()` checked → if still `"playing"`, proceed to computer turn

### Computer Turn

8. `isProcessing = true`, board `pointer-events: none` applied
9. Random delay 300–800ms (`setTimeout`) simulates thinking
10. `game.computer_move()` → WASM call → `get_computer_move()` in `src/ai.rs` (minimax + 25% random mistake) → applies move internally → returns chosen position as `u8`
11. `sounds.computerMove()`, `renderBoard()`, `isProcessing = false`
12. `game.get_status()` checked → if game over, `handleGameOver()`

### Game Over

13. `game.get_winning_positions()` → `Uint32Array[3]` (or empty)
14. `renderBoard(winPositions)` adds `cell--winning` CSS class to winning cells
15. `showWinLine(positions)` maps position key to CSS class on `#win-line` overlay
16. Score updated in memory, saved to `localStorage`, score DOM elements patched
17. `restartBtn.hidden = false`

### Reset

18. `game.reset()` → WASM call → `Game::new()` inside WASM memory
19. `boardEl.innerHTML = ''` forces full cell rebuild on next `renderBoard()`
20. Win-line overlay hidden, restart button hidden, status reset to "Your turn"

## Key Abstractions

**`WasmGame` (Rust struct / JS class):**
- Purpose: Opaque game handle — the single shared object between JS and Rust
- Rust: `src/wasm_api.rs` — wraps `Game`, adds `#[wasm_bindgen]` annotations
- JS: `pkg/tic_tac_toe.js` — generated class that holds a pointer into WASM linear memory
- Pattern: Handle/opaque-pointer — JS cannot inspect or modify internals directly

**`Game` (Rust struct):**
- Purpose: Canonical board state and move logic
- Location: `src/board.rs`
- Pattern: Value-type state machine (`InProgress` → `Won`/`Draw`); immutable reads, explicit mutation via `make_move()`

**`GameStatus` (Rust enum):**
- Purpose: Encodes terminal/non-terminal state with winner + positions when won
- Location: `src/board.rs`
- Values: `InProgress`, `Won { winner: Player, positions: [usize; 3] }`, `Draw`

**`get_computer_move()` (Rust fn):**
- Purpose: AI move selection — minimax with 25% random mistake rate
- Location: `src/ai.rs`
- Pattern: Pure function (takes `&Game`, returns `Option<usize>`) — stateless, no side effects; uses `rand::rng()` for RNG

## Entry Points

**Browser Game:**
- HTML: `index.html` loaded by browser
- JS: `src/main.js` (ES module, `<script type="module">`) — `main()` async function runs on page load
- WASM: `pkg/tic_tac_toe.js` + `pkg/tic_tac_toe_bg.wasm` fetched and instantiated by `init()`

**CLI Game (development/demo):**
- Location: `src/main.rs`
- Triggers: `cargo run`
- Responsibilities: Terminal rendering, stdin input loop, same `Game` + `get_computer_move()` as browser

**WASM Test Page:**
- Location: `test.html`
- Purpose: Manual browser-based WASM bridge smoke test (13 assertions)
- Not linked from `index.html`; accessed directly at `/test.html` during dev

## Error Handling

**Strategy:** Errors are contained at boundaries; WASM panics surface to browser console; JS errors show an overlay.

**Patterns:**
- Rust: `Game::make_move()` returns `Result<(), String>` — `WasmGame::make_move()` converts to `bool` for JS
- Rust: `#[wasm_bindgen(start)]` registers `console_error_panic_hook` so panics print to browser console with backtrace
- JS: WASM init failure caught by `main().catch()` → DOM replaced with `.error-overlay` div (safe `textContent`, no XSS)
- JS: `localStorage` read/write wrapped in try/catch to handle `SecurityError` in private browsing

## Cross-Cutting Concerns

**Score Persistence:**
- localStorage key `ttt-score` stores `{ wins, losses, draws }` JSON
- Loaded on module init in `src/main.js`, saved after every game over

**Mute Preference:**
- localStorage key `ttt-muted` stores `"true"` / `"false"`
- Loaded on `audio.js` module init (IIFE), toggled by mute button

**Accessibility:**
- ARIA roles (`role="grid"`, `role="gridcell"`, `role="button"`, `aria-live="polite"`)
- Keyboard navigation: Enter/Space on focused cells trigger `handleCellClick`
- `tabIndex` removed from taken cells; added to empty cells
- Win-line overlay has `aria-hidden="true"`

**Animations:**
- CSS only — no JS animation code
- `cell-pop` keyframe on `.cell--x`, `.cell--o` (piece placement bounce)
- `win-draw` / `win-draw-col` / `win-draw-diag-*` keyframes on `.win-line` variants
- All animations respect `prefers-reduced-motion`

**Double-click Guard:**
- `isProcessing` boolean in `src/main.js` blocks human clicks during the computer's async turn
- `thinkingTimer` ID stored so reset can cancel a pending computer delay
